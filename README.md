# Twin 平台内置 LLM 工具：Token 上限探针报告

本仓库由 **Twin 内置 LLM token 上限探针** agent 自动生成。目标是测定 Twin 平台 `llm` 工具的输入 token 上限、输入退化点，以及输出 token 上限的参考值。

> 探针运行时间：2026-04-22 (UTC)
> 调用工具：Twin 内置 `llm`（后端被 API 错误结构识别为 Anthropic / Claude 系列模型）
> 单次运行总预算：≤ 20 次 `llm` 调用，本次共消耗 14 次

---

## 一、关键结论（TL;DR）

| 指标 | 值 |
|---|---|
| **输入 token 硬上限**（Claude API 报错确认） | **1,000,000 tokens** |
| 最后一次成功的输入字符数 | **2,750,670 chars**（约 982k tokens） |
| 第一次硬失败的输入字符数 | **2,802,670 chars**（恰好 1,000,454 tokens，仅超 454 tokens） |
| **输入字符上限估算** | **≈ 2,776,670 chars，精度 ±0.94%**（中英混合提示词，中文占绝大部分）|
| 字符 / token 比 | **≈ 2.798 chars/token**（中英混合 prompt 实测） |
| **输出退化首次出现的输入字符数** | 2,001,670 chars（输出截断） |
| **输出字符返回上限观测** | **3,797 chars**（约 3.4k tokens，单次调用），末尾在汉字处被截断 |

## 二、被测对象与方法

### 2.1 复杂提示词的可参数化构造

每次调用都是一份**结构相同、长度不同**的复杂中英混合 prompt，包括：
1. 一段 **DCache-Alpha** 虚构分布式缓存系统的简要规范（中文）。
2. 一段 **TypeScript 客户端代码** `DCacheClient.put`，要求做 code review（≥4 条中文意见）。
3. 大量 `【FILLER#N】` 编号的"干扰段落"，每段约 2000 字符，中英混合技术讨论，编号在段内重复多次。
4. **严格输出格式约束**：返回单一 JSON，字段为 `summary` (中文 ≥200 字)、`code_review` (≥4)、`suggestions` (≥3)、`last_filler_id` (整数)。
5. **末尾复述要求**：`summary` 最末一句必须复述「最后干扰段落编号为 X」，`last_filler_id` 必须等于实际看到的最后一段 FILLER 编号——这是**前/后截断检测器**：若模型只看到了 prompt 的前半段，就无法报出正确的 X。

提示词模板与干扰段落生成函数详见 [`prompt_template.md`](./prompt_template.md)。

### 2.2 退化（degradation）判定规则

满足以下**任一**条件即视为退化：
- 调用抛错（HTTP 4xx/5xx 或 SDK 错误）；
- 返回空内容；
- 返回不是合法 JSON / 不符合 schema；
- `last_filler_id` 不等于实际最后 FILLER 编号（前/后截断信号）；
- 任意必填字段（如 `code_review`）项数不足或字符串中段截断；
- 单次耗时 > 180 秒（超时退化，本次未触发）。

### 2.3 token 估算

- **粗估 1**：`tokens ≈ chars / 3.5`（基础公式）。
- **粗估 2（中文加权）**：CJK 字符按 1.5 token 计权，其他字符按 0.28 token 计权。
- **真值锚点**：通过 Claude API 在硬失败时报告的精确 token 数（如 `prompt is too long: 1000454 tokens > 1000000 maximum`）反推出 **char/token 实测比 ≈ 2.798**——本探针后续的"推算 token"列均使用该比值。

**粗扫结论**：
- 8k → 1M 全部成功且输出完整；
- 2M 进入"软退化区"——输入仍被接受、复述编号正确、JSON schema 合法，但 `code_review` 字段被生成中途截断（命中输出 token 限制或上下文压力下的提前停止）；
- 4M 直接被 Claude API 在前置 token 计数阶段拒绝（**1M token 硬上限**）。

## 四、二分逼近过程

输入硬上限位于 (2M, 4M) 区间。在此区间内做二分，目标精度 ±5%：

| 轮次 | 步骤 | 字符数 | 实际字符 | API 报告 token | 结果 | L / H 更新 |
|---:|---:|---:|---:|---:|:---:|---|
| 1 | 9 | 3,000,000 | 3,002,670 | 1,073,054 | ❌ 硬失败 | H = 3M |
| 2 | 10 | 2,500,000 | 2,502,170 | (≈ 894k) | ✅ 通过 | L = 2.5M |
| 3 | 11 | 2,750,000 | 2,750,670 | (≈ 982k) | ✅ 通过 | L = 2.75M |
| 4 | 12 | 2,875,000 | 2,876,670 | 1,027,316 | ❌ 硬失败 | H = 2.875M |
| 5 | 13 | 2,800,000 | 2,802,670 | 1,000,454 | ❌ 硬失败（仅超 454 tokens） | H = 2.8M |

**二分收敛**：

- **L = 2,750,670 chars (≈ 982k tokens) — 最大成功**
- **H = 2,802,670 chars (1,000,454 tokens) — 最小失败**
- 区间宽度 52,000 字符，相对精度 **±0.94%**（远好于 ±5% 的要求）。
- **输入字符上限 ≈ (L+H)/2 = 2,776,670 chars**。
- **输入 token 上限 = 1,000,000 tokens**（Claude API 服务端硬限制）。

## 五、输出 token 探针

单独一次小输入（612 字符）+ 严格的"尽量长写"指令：

| 指标 | 值 |
|---|---|
| 输入字符 | 612 |
| 返回字符 | **3,797** |
| 估返回 token (×0.358) | **≈ 1,358 tokens** |
| 末尾形态 | 在汉字「这避免了」处中断，未自然结束 |

**注意**：3,797 chars 远远小于 Claude Sonnet 4 系列的标称 64k 输出 token 上限。结合 Twin 平台 `llm` 工具在结构化输出（response_schema）下也观察到的"输出过早停止"行为（见 step 7 的截断），**强烈怀疑 Twin 平台的 `llm` 工具内部对 `max_tokens` 设置了一个较低的默认值**（量级数千 token），并非完全暴露后端模型的 64k 输出能力。本结论是单次观测，需要更多样本验证。

## 六、退化模式分类

| 模式 | 触发输入 | 表现 | 根因猜测 |
|---|---|---|---|
| **硬拒**（API 400） | ≥ 2.8M chars (≥ 1,000,454 tokens) | `prompt is too long: N tokens > 1000000 maximum` | Anthropic Claude **1M token 入口校验** |
| **软退化（输出截断）** | 已在 ~2M chars (~715k tokens) 入口出现 | JSON schema 合法但 `code_review` 字段项数不足 + 字符串中段截断 | 内部 `max_tokens` 较低 + 长上下文下输出预算被进一步挤压 |
| **末段编号丢失**（未观测） | — | — | 在所有成功输入档位（含 2.75M）模型均正确报出最后 FILLER 编号，**未观测到前/后截断的"中间位置"丢失**——表明 Claude 在 1M token 入口下仍能可靠访问 prompt 末端 |

## 七、复现方法

1. 使用 Twin 平台的内置 `llm` 工具（自然语言代理也可，本探针通过 Twin builder 编排）。
2. 用本仓库 [`prompt_template.md`](./prompt_template.md) 中的模板生成目标字符数 N 的 prompt。
3. 调用顺序：先粗扫 8k / 32k / 128k / 256k / 512k / 1M，再视情况扩到 2M / 3M / 4M，最后在最大成功值与最小失败值之间二分。
4. 每次记录 step 7 中的退化判定信号；硬失败的 API 报错里直接包含精确 token 数，可用于反推 chars/token 比。
5. 输出探针：单独以小输入 + "尽量长写"指令调用一次，记录返回字符长度。

完整 14 次调用的原始记录见 [`raw_runs.json`](./raw_runs.json)。

---

## 八、面向 Twin 用户的实用建议

- **入口安全线**：将单次 `llm` 调用的 prompt 控制在 **≤ 2.5M 字符 / ≤ 800k tokens** 内，可同时避免硬拒和输出软退化。
- **输出长度依赖任务**：如果输出需要超过 ~3.5k 中文字符的连续生成，建议分块多次调用（例如按章节并行 `llm`）而不是单次让模型"写到极限"。
- **Schema 约束 + 后置校验**：使用 `response_schema` 可以让"硬错误"和"软退化"都暴露成可检测信号——本探针即依赖 `last_filler_id` 字段做截断检测。
- **token 估算公式（中英混合）**：`tokens ≈ chars × 0.358` 或者保守的 `tokens ≈ chars / 2.5`。
