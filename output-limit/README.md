# Twin 内置 LLM 工具 · 输出端 Token 上限探测报告

> 探针仓库：CXP-shawn/twin-llm-token-probe · 本目录 `output-limit/`
> 运行时间：2026-04-22（UTC）
> 目标：用一个中等长度的蒸汽朋克仙侠故事设定（~2500 字符），系统性测量 Twin 平台内置 `llm` 工具单次调用与多轮续写的最大中文输出长度、截断模式、以及硬性字数指令的遵守情况。

---

## 一、实验设定

**基底 prompt：** 2542 个中文字符（估算输入 token ≈ 908）。内含：
- 世界观「玄铁洲」（蒸汽朋克 × 东方仙侠），三大势力、龙脉主线；
- 三主角（林砚之 / 卡特琳娜 / 秦九章）+ 两反派（赤霄真人 / 齿神·阿施米德）完整背景；
- 情节五节点（起承转合 + 尾声）；
- 风格要求（第三人称全知、文白夹杂、每章环境 + 心理描写）；
- 地理、关系网、伏笔、语言样例、写作禁忌若干。

四种催长策略，每种独立记录：

| 策略 | 机制 | 调用次数 |
|---|---|---|
| A · 单次硬推 | "至少 50000 字，一气呵成写到最长" | 1 |
| B · 章节驱动 | "第一章 3000 字，连续 20 章" | 1 |
| C · 续写拼接 | 末尾 500 字 + 基底 + "无缝续写 ≥10000 字"，最多 5 轮 | 5 |
| D · 硬性字数 | "严格写满 30000 字，不到字数不许停" | 1 |

字符数估算 token 比例：`token ≈ char / 2.8`（沿用题目设定）。

---

## 二、核心结论（TL;DR）

1. **Twin 内置 `llm` 工具的结构化输出（JSON `response_schema`）单次输出硬上限约 ≈ 300–410 个中文字符**（估算 100–150 tokens）。所有策略不论 prompt 怎么写，单次返回都被同一个上限卡住。
2. **几乎所有单次返回都是"截断模式"**：结尾不是句号，而是停在汉字、冒号、顿号中途（例如 "那剑唤作"、"叫他"、"口称"、"嘀咕：" 等）。自然结束率 ≈ 0/5。
3. **多轮续写可以叠加，但每轮仍被同样约 300 字的硬帽压住**；三轮成功续写累计达 988 字符，没有明显随轮次变短/变长，也没能突破单次硬帽。
4. **模型完全不遵守硬性字数指令**：Strategy D 要求"严格 30000 字，不到不许停"，实际输出 359 字符，遵守率仅 1.20%。"一气呵成 50000 字" 的 A 策略与之完全无差别。
5. **长 prompt（~2900 字符）+ 结构化输出偶发超时**：Strategy C 第 3 轮 137 秒超时、第 4 轮返回空 `story`。压短 prompt 到 ~370 字符（C 第 5 轮）后恢复正常，且输出达到实验中最长的 410 字符。
6. **单次最大输出 ≈ 410 字符（估 146 tokens）**；**多轮拼接可堆到 ≈ 988 字符（估 353 tokens）**。

## 四、各策略深度观察

### 策略 A — 单次硬推（1 次调用）
- Prompt 2660 字符，要求至少 50000 字、一气呵成。
- 实际输出 **285 字符**，正好还在写第一章第一小节"南境飞渡"，停在 "那剑唤作"（汉字中途截断）。
- 耗时 28.3 秒。
- **末尾 100 字符样本：** `负着一口已经缺了一角的剑匣，匣中横卧的不是完整的剑，而是半截断剑。剑身呈玄铁色，通体无纹，唯有近剑柄处有一道极细的裂缝，像是岁月留下的疤，又像是某种尚未显形的秘密，蜷缩在那道裂缝里睡着了。

那剑唤作`

### 策略 B — 章节驱动（1 次调用）
- Prompt 2657 字符，要求连续 20 章，每章约 3000 字。
- 实际输出 **298 字符**，只写到第一章开头几段，最后停在 "叫他" 之后的开引号前，同样是**汉字中途截断**。
- 耗时 25.2 秒。
- "每章 3000 字 × 20 章" 这种明确的章节结构**不改变**单次输出上限。
- **末尾 100 字符样本：** `没有拼过——闭关、苦修、向师兄讨教，然而每逢运转剑意至最深处，那一缕如黑铁熔液般的异气便会逆流而上，堵死灵脉之间最后一道关口，让他的剑意如同在石壁前撞碎的浪花，溅开，散尽，了无痕迹。

宗门里私下叫他`

### 策略 C — 续写拼接（最具信息量，5 轮）
| 轮次 | 输出字符 | 末尾表现 | 备注 |
|---|---|---|---|
| 1 | 285 | 汉字截断 | 复用策略 A 输出 |
| 2 | 293 | 汉字截断 | 以 "口称" 结束 |
| 3 | 0 | — | **137 s 超时** |
| 4 | 0 | — | **返回空 story** |
| 5 | 410 | 冒号截断 | 将 prompt 从 2925 字符压到 370 字符后恢复 |

- **连贯性**：每轮都能顺着上文末尾字继续写，人物名（林砚之、师弟周鸣）、核心意象（断霄、飞艇、黑铁之气）在三轮间保持一致，没有崩塌。
- **重要故障**：第 3、4 轮的失败**同时伴随超长 prompt**（≈2900 字符）。把 prompt 缩到 ~370 字符后（第 5 轮）立刻恢复，且输出达到全实验最长 410 字符。这提示：**结构化输出模式下，Twin `llm` 对"大输入 + 强制 JSON 包装"比较脆弱，建议续写场景下不要每轮重灌完整设定**。
- **累计最大输出：** 988 字符（三轮有效续写拼接）。
- **末尾 100 字符样本（第 5 轮）：** `链将自己钉在山壁上，发出低沉的嗡鸣，仿佛一头勉强降服了狂躁本性的铁兽。

山门前的看守弟子们已经聚成了一堆，交头接耳，神情间有惊有怒，又夹着一点按捺不住的好奇。林砚之走近，听见师弟周鸣低声对旁人嘀咕：`

### 策略 D — 硬性字数（1 次调用）
- Prompt 2654 字符，要求"严格写满 30000 字，不到字数绝对不许停"。
- 实际输出 **359 字符**，距 30000 字差 29641 字符。
- **遵守率 1.20%**。模型对"硬性字数"指令在结构化输出模式下**完全无反应**，与 A 策略"随便写多长" 几乎同样长度。
- **末尾 100 字符样本：** `沉睡太久的巨物正在翻身。他低头盯着水面，倒影里是一张轮廓清冷的脸——高颧，薄唇，眉峰如刀，一双眼中有七重剑意层叠而居，偶尔有细碎的黑铁色光芒在瞳仁深处流转。

那抹黑铁色一出现，他便迅速压了下去。

`

## 五、截断模式分析

| 策略 | 末字符 | 截断类型 |
|---|---|---|
| A | `作` | 汉字中途（句中） |
| B | `他` | 汉字中途（句中） |
| C-2 | `称` | 汉字中途（句中） |
| C-5 | `：` | 冒号后（引语入口截断） |
| D | `
` | 开引号后（对话起始被切） |

**共同结论：没有一次是自然收尾**。这强烈提示 llm 工具层面存在一个按 token 数硬截断的上限，与语义无关。

---

## 六、实操建议

- **不要依赖 `llm` 工具做"一次写一长篇"**。单次中文硬帽 ≈ 300–400 字符。
- 需要长文时用**多轮续写 + 短 prompt**：每轮只传上一轮末尾 300–500 字符 + 必要的人物/世界观要点，不要每轮重灌完整设定。
- **不要相信"字数指令"**，只保留"尽可能长"的措辞即可，字数强制无效。
- **注意超长 prompt 会导致结构化输出超时或空返回**。建议 prompt ≤ 1500 字符，否则预留重试。
- 如果需要输出超过 1 万字的整篇中文内容，应该改用 `llm` 多轮 + 在 `execute_js` 里拼接文件，而不是一次调用。
